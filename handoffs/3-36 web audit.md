0. Scope & source-of-truth notes (read first)

Two framing issues change how every finding below should be read:

A. The audit brief assumes Astro. This project is Next.js. Stack is Next.js 16 (App Router) + React 19 + Tailwind v4 + shadcn-style primitives + next-themes + lucide-react. There is no Astro. "Astro Code Quality" is re-scoped to Next.js best practice. This is not a defect in the site — it's a defect in the brief.

B. The brief's "Required Academic Content" list conflicts with the project's own approved docs. The brief demands Certificates, Seminars, per-year (1st–4th) software catalogs, GUI screenshots for every project, and Capstone hardware breakdowns. The PRD (docs/03_PRD.md) — which the brief itself names as the #1 source of truth — deliberately scopes Version 1 to exactly 5 pages (Home, About, Projects, Skills, Contact) and explicitly defers Case Studies, Additional Project Metadata, and Certifications to "Future Enhancements" (§16), listing them as out of scope (§15).

So there are two different, contradictory rulebooks. I score against both and flag the conflict as the single most important decision you must make — because it's the one thing that can fail you academically regardless of build quality. You need to confirm with your professor whether the school's portfolio rubric (certs/seminars/per-year MPs/screenshots/hardware) is mandatory. If it is, your PRD scoped out required deliverables and the site is academically incomplete no matter how well engineered. If the PRD is the accepted contract, the site is nearly compliant. I cannot resolve this from the repo — it depends on your program's rubric.

2. Severity-Ranked Findings

Critical

1. Contact email is internally contradictory — possibly a dead address. lib/constants/contact.ts:6 defines CONTACT_EMAIL = "carlieendaya@gmail.com", but the visible contact card and its mailto: (lines 46–47) and the footer (profiles.ts:12) all use carlieshaye.endaya@gmail.com — a different address. Your registered account email is carlieendaya@gmail.com. The address recruiters will actually click is the other one. If that inbox doesn't exist, every "email me" path is broken. Violates CONTENT-004/005 (consistent, non-contradictory content). Confirm which address is real and make all four locations identical. (This is the one finding I need you to resolve — I can't verify which inbox is live.)

High

2. No custom 404 page. No app/not-found.tsx (or error.tsx). TECH-013 requires "a user-friendly error page that provides a clear path back to primary navigation." Next.js ships a bare default with no header/footer/nav — it fails the "path back" requirement. A mistyped URL dead-ends the visitor.
3. Zero screenshots anywhere. No project (incl. capstone) renders a GUI image; public/images is empty. Fails the academic brief outright, and contradicts your own Design System component inventory. For a systems/backend portfolio this is defensible as a philosophy — but a capstone with no visual evidence is a credibility gap to most evaluators.

Medium

4. Footer external links aren't marked external. site-footer.tsx:37–44 renders GitHub/LinkedIn/Email as plain <a> — no new-tab, no rel="noopener", no external icon, styled identically to internal page links. Meanwhile the correct ExternalLink component (icon + underline + "opens in a new tab" SR text + rel) is used on the Contact page. Inconsistent, and violates NAV-013 / A11Y-011 / TECH-029 (external destinations must be distinguishable).
5. SITE_URL defaults to http://localhost:3000. site.ts:27. If NEXT_PUBLIC_SITE_URL isn't set at deploy, metadataBase, canonical, and OpenGraph URLs all point at localhost — broken social cards and SEO. Set the env var before shipping.
6. Capstone hardware not surfaced as hardware. Pi/Wemos are buried in a flat technologies[]. Rubric (and good IoT presentation) wants a distinct Hardware grouping.

Low
